# my_ucrt_star 项目开发历程与问题分析

## 📊 项目概览

**总提交数**: 53 次  
**开发周期**: 2024年12月 ~ 2025年9月  
**项目类型**: RISC-V 虚拟化安全操作系统开发

**核心模块**:
- 自定义 QEMU 机器 (quard-star)
- OpenSBI 1.2 移植
- Trusted Domain (FreeRTOS)
- Untrusted Domain (自定义 OS - timeros)

---

## 📋 项目架构概览

```
启动流程: MROM → LowLevelBoot → OpenSBI → [Trusted Domain (CPU7, FreeRTOS) + Untrusted Domain (CPU0-6, OS)]
                    ↓
            设备树配置多域隔离
```

### 第一阶段：QEMU 机器创建 (2024.12.29)
**提交**: `832a8fe2d` - "1.Qemu中自定义开发板"

#### 遇到的问题：QEMU 机器类型注册失败

**如何发现**:
```bash
# 运行 QEMU 时报错
Unknown machine type: quard-star
```

**根本原因**:
1. 需要在 `hw/riscv/meson.build` 中添加新机器的编译规则
2. 需要在 `configs/devices/riscv64-softmmu/default.mak` 中启用新设备
3. `hw/riscv/Kconfig` 需要添加配置选项

**解决方案**:
```c
// qemu-8.0.2/hw/riscv/quard_star.c
static const TypeInfo quard_star_machine_typeinfo = {
    .name       = MACHINE_TYPE_NAME("quard-star"),
    .parent     = TYPE_MACHINE,
    .class_init = quard_star_machine_class_init,
    .instance_init = quard_star_machine_instance_init,
    .instance_size = sizeof(QuardStarState),
};
type_init(quard_star_machine_init_register_types)
```

**经验总结**:
- QEMU 的机器类型注册使用 QOM (QEMU Object Model) 框架
- 需要在多个构建配置文件中同步添加条目

### 第二阶段：底层启动代码开发 (2024.12.29 ~ 2025.1.6)
**提交**: `238bc0d8a` - "7.为my_ucrt_star移植opensbi-1.2"

#### 遇到的问题 2.1：从裸机串口到加载 OpenSBI

**初始代码** (简单串口输出):
```asm
// 最初的 start.s - 只能输出 "Hello Quard Star board!"
_start:
    csrr    a0, mhartid
    li      t0, 0x10000000
    li      t1, 'H'
    sb      t1, 0(t0)
    // ... 重复输出字符
```

**问题**: 只能做最简单的串口输出，无法加载复杂固件

**解决方案**: 实现数据加载宏
```asm
// 改进后的 start.s
.macro load_data,_src_start,_dst_start,_dst_end
bgeu    \_dst_start, \_dst_end, 2f
1:
    lw      t0, (\_src_start)
    sw      t0, (\_dst_start)
    addi    \_src_start, \_src_start, 4
    addi    \_dst_start, \_dst_start, 4
    bltu    \_dst_start, \_dst_end, 1b
2:
.endm
```

#### 遇到的问题 2.2：多核启动不同步

**如何发现**:
- 只有 hart 0 能正常启动
- 其他 hart 启动时 OpenSBI 还未初始化完成

**根本原因**:
非 boot hart 没有等待 OpenSBI 初始化完成就尝试跳转

**解决方案**:
```asm
csrr    a0, mhartid
li      t0, 0x0     
beq     a0, t0, _no_wait
loop    0x1000      // 非0核固定延时等待
_no_wait:
    // 继续执行
```

**经验总结**:
- 多核启动需要同步机制
- 简单的忙等待在开发阶段足够，但生产环境需要更可靠的同步原语

### 第三阶段：OpenSBI Domain 机制实现 (2025.1.11)
**提交**: `8b20edfa9` - "8.基于opensbi为my_ucrt_star创建domain"

#### 遇到的问题 3.1：设备树配置错误导致 Domain 初始化失败

**如何发现**:
```
OpenSBI 启动日志显示:
Domain0: boot hart id not found
```

**根本原因分析**:
设备树中 `opensbi-domains` 节点配置错误：
```dts
// 错误的配置
tdomain: trusted-domain {
    possible-harts = <&cpu7>;
    // 缺少 boot-hart 字段
};
```

**解决方案**:
```dts
// 正确的配置
tdomain: trusted-domain {
    compatible = "opensbi,domain,instance";
    possible-harts = <&cpu7>;
    boot-hart = <&cpu7>;          // 必须指定 boot hart
    next-addr = <0x0 0xBF800000>; // Trusted Domain 入口
    next-mode = <0x1>;            // S-Mode
    regions = <&tmem0 0x7>, ...;  // 内存区域权限
};
```

#### 遇到的问题 3.2：内存区域权限配置错误

**如何发现**:
- Trusted Domain 可以访问 Untrusted Domain 内存（应该被禁止）
- 或反之，导致安全隔离失效

**根本原因**:
```dts
// 错误的权限配置 - allmem 覆盖了整个地址空间
allmem: allmem {
    compatible = "opensbi,domain,memregion";
    base = <0x0 0x0>;
    order = <64>;  // 2^64 = 整个地址空间
};

udomain: untrusted-domain {
    regions = <&tmem0 0x0>, ..., <&allmem 0x7>; // 0x7 = RWX 权限
};
```

**解决方案**:
- 移除 `allmem` 的 RWX 权限
- 明确指定每个 Domain 可以访问的内存区域

### 第四阶段：OS 基础功能开发 (2025.1.12 ~ 2025.2.8)

- `cb1251d24` - 封装 printf 函数

#### 遇到的问题 4.1：SBI 调用返回错误

**如何发现**:
```c
sbi_console_putchar('H');  // 无输出
```

**根本原因**:
SBI 调用约定错误：
```c
// 错误的调用方式
asm volatile (
    "mv a0, %0\n\t"    // ext 放在 a0
    "mv a1, %1\n\t"    // fid 放在 a1
    "ecall"
    :: "r"(ext), "r"(fid)
);
```

**正确的调用约定**:
```c
// 正确的调用方式 - SBI 规范要求
asm volatile (
    "mv a7, %0\n\t"    // ext 放在 a7
    "mv a6, %1\n\t"    // fid 放在 a6
    "ecall"
    :: "r"(ext), "r"(fid)
);
```

**经验总结**:
- RISC-V SBI 规范规定：a7=extension ID, a6=function ID
- 返回值在 a0 (error code), a1 (value)

#### 遇到的问题 4.2：编译环境问题 - 缺少 compile_commands.json

**如何发现**:
- VSCode clangd 无法正确索引代码

### 第五阶段：Trap 机制与上下文切换 (2025.1.26)
**提交**: `bc4bc09b9` - "11.实现U模式的trap机制"

#### 遇到的问题 5.1：用户态程序无法正确触发 syscall

**如何发现**:
- 执行 `ecall` 指令后系统挂起
- 没有进入 trap handler

**根本原因**:
1. `stvec` 寄存器未正确设置
2. trap handler 地址错误
3. sstatus.SPP 位未正确设置

**解决方案**:
```c
// 正确设置 trap 入口
void trap_init() {
    w_stvec((reg_t)trap_handler);  // 设置 trap 向量表
}

// 进入用户态前设置 SPP
reg_t sstatus = r_sstatus();
sstatus &= ~(1L << 8);  // SPP = 0, 表示从 U-Mode 返回
w_sstatus(sstatus);
```

#### 遇到的问题 5.2：上下文保存/恢复不完整

**如何发现**:
- 从 trap 返回后寄存器值错乱
- 程序执行异常，PC 跳转错误

**根本原因**:
上下文结构体定义不完整，缺少某些寄存器：
```c
// 最初的不完整定义
typedef struct pt_regs {
    reg_t ra, sp, gp, tp;
    // 缺少 x0, t0-t2, s0-s11 等
} pt_regs;
```

**解决方案**:
```c
// 完整的上下文定义
typedef struct pt_regs {
    reg_t x0;       // zero
    reg_t ra;       // return address
    reg_t sp;       // stack pointer
    reg_t gp;       // global pointer
    reg_t tp;       // thread pointer
    reg_t t0-t2;    // temporary
    reg_t s0-s11;   // saved registers
    reg_t a0-a7;    // arguments
    reg_t s2-s11;   // more saved registers
    reg_t t3-t6;    // more temporary
    reg_t sepc;     // exception program counter
    reg_t sstatus;  // status register
} pt_regs;
```

### 第六阶段：MMU 与地址空间 (2025.2.12)
**提交**: `46182d3c0` - "17.开启mmu实现虚实地址映射"

#### 遇到的问题 6.1：开启 MMU 后立即崩溃

**如何发现**:
```c
w_satp(satp_value);  // 开启 MMU 后 PC 指向无效地址
```

**根本原因**:
1. 页表映射不完整
2. 没有映射当前正在执行的代码地址
3. TLB 未刷新

**解决方案**:
```c
// 必须映射的地址范围:
// 1. 内核代码段
// 2. 内核数据段  
// 3. 栈空间
// 4. 设备寄存器 (UART, CLINT, PLIC)

// 开启 MMU 后刷新 TLB
asm volatile("sfence.vma zero, zero");
```

#### 遇到的问题 6.2： Sv39 页表格式错误

**如何发现**:
- 页表遍历失败，触发 page fault
- 物理地址与虚拟地址映射错误

**根本原因**:
页表项格式错误，混淆了 PPN 和 PFN：
```c
// 错误的页表项创建
uint64_t pte = (pa >> 12) << 10;  // 错误：没有设置权限位
```

**解决方案**:
```c
// 正确的页表项创建
#define PTE_V (1L << 0)  // Valid
#define PTE_R (1L << 1)  // Read
#define PTE_W (1L << 2)  // Write
#define PTE_X (1L << 3)  // Execute

uint64_t pte = ((pa >> 12) << 10) | PTE_V | PTE_R | PTE_W | PTE_X;
```

**如何发现**:
- 切换任务后函数调用异常

**根本原因**:

### 第八阶段：Fork/Exec 实现 (2025.2.17)
**提交**: `edf80455c`, `efd8be0b2` - "sys-fork/exec的实现"

#### 遇到的问题 8.1：Fork 后子进程返回地址错误

**如何发现**:
- 子进程没有从 syscall 返回
- 或返回到了错误的地址

**根本原因**:
Trap 上下文中的 `sepc` 没有正确设置：
```c
// fork 时复制父进程的 trap 上下文
memcpy(child_trapframe, parent_trapframe, sizeof(TrapContext));

// 子进程应该返回 0
child_trapframe->a0 = 0;  // 子进程返回值
// 但 sepc 保持不变，会重新执行 syscall 指令
```

**解决方案**:
```c
// sepc 在 trap handler 中已经 +4 (跳到下一条指令)
// 所以子进程的 sepc 已经是正确的返回地址
// 只需要设置返回值
child_trapframe->a0 = 0;
```

#### 遇到的问题 8.2：Exec 后内存泄漏

**如何发现**:
- 多次 exec 后物理内存耗尽
- 旧程序的页表没有释放

**解决方案**:
```c
int exec(const char* name) {
    // 保存旧页表
    PageTable old_pagetable = proc->pagetable;
    
    // 创建新页表
    proc_pagetable(proc);
    
    // 加载新程序
    load_segment(ehdr, proc);
    
    // 释放旧页表
    proc_freepagetable(&old_pagetable, oldsz);
    
    // 设置新的执行上下文
    trapframe->sepc = ehdr->e_entry;
}
```

## 🔴 核心问题与神坑汇总

### 难题一：多核启动时序与同步问题

#### 问题描述
OpenSBI 初始化 8 个 HART，但不同域的启动地址不同（Trusted: 0xBF800000, Untrusted: 0x80200000）。**非启动核（non-boot hart）可能在固件还没准备好时就尝试跳转**。

#### 如何发现
- 串口输出只有 CPU0 或 CPU7 的打印，其他核无输出
- QEMU 挂起，需要 GDB 连接到不同 HART 查看 PC 寄存器
- OpenSBI 日志显示 `hart X: domain startup failed`

#### 根本原因

```asm
; boot/start.s 中的忙等待代码
csrr    a0, mhartid
li      t0, 0x0     
beq     a0, t0, _no_wait
loop    0x1000    ; 非0核固定延时，可能不够
```

非 boot hart 的延时固定，如果 OpenSBI 初始化慢于延时，会导致跳转时目标代码未就绪。

#### 解决方案

1. **使用 OpenSBI 提供的 HART 间同步机制**（IPI 或共享内存标志位）

2. **在目标代码入口处设置同步屏障**：
   ```c
   // 每个域的入口代码
   static volatile int ready_flag = 0;
   
   void domain_entry() {
       if (hartid == boot_hart) {
           // 初始化代码...
           __sync_synchronize();
           ready_flag = 1;
       } else {
           while (!ready_flag) { wfi(); }
       }
   }
   ```

### 难题二：Trust Domain 内存隔离失效

#### 问题描述
Trusted Domain (0xBF800000-0xBFFFFFFF) 和 OS 内存（0x80000000-0xC0000000）有重叠区域。**OS 可能意外访问或覆盖 Trusted Domain 内存**。

#### 如何发现
- Trusted Domain 随机崩溃或数据损坏
- QEMU 内存检查器显示非预期的写操作
- OpenSBI 触发 PMP（Physical Memory Protection）异常

#### 根本原因

查看设备树配置：

```dts
// quard_star_sbi.dts
udomain: untrusted-domain {
    regions = <&tmem0 0x0>, <&tmem1 0x0>, ... <&allmem 0x7>;
    // allmem 覆盖整个 0x0-0xFFFFFFFFFFFFFFFF，权限 0x7(RWX)
};
```

`allmem` 的权限设置允许 Untrusted Domain 访问所有内存，**隔离策略失效**。

#### 解决方案

1. **修正设备树中的内存区域权限**：
   ```dts
   udomain: untrusted-domain {
       regions = <&tmem0 0x0>, <&tmem1 0x0>, ...; // 排除 trusted 区域
       // 不要包含 allmem，或将其权限设为 0
   };
   ```

2. **启用 PMP 检查**，在 OpenSBI 配置中确保 `PMP_GRANULARITY` 设置正确

### 难题三：PLIC 中断路由冲突

#### 问题描述
UART2（Trusted Domain 专用串口）的中断可能被错误地路由到 Untrusted Domain，或反之。

#### 如何发现
- Trusted Domain 的串口无中断响应（查询方式可以工作）
- PLIC 寄存器读取显示中断被错误核心认领
- 使用 QEMU `-d int` 查看中断分发日志

#### 根本原因

设备树中所有 UART 都连接到同一个 PLIC，但 Trusted/Untrusted Domain 的中断上下文是独立的：

```dts
plic@c000000 {
    interrupts-extended = <&cpu0_intc 11>, <&cpu0_intc 9>, ...>; // 所有 CPU
};
```

#### 解决方案

1. **为 Trusted Domain 配置独立的中断控制器**（如 APLIC/IMSIC，需要 QEMU 支持）

2. **在软件层面过滤中断**：在 trap handler 中检查当前 domain，忽略不属于本 domain 的中断

3. **修改 QEMU 的 PLIC 实现**，支持 domain-based 中断屏蔽（需要修改 `hw/intc/sifive_plic.c`）

### 难题四：OpenSBI Domain 切换失败

#### 问题描述
调用 `sbi_domain_reset()` 或尝试从 Untrusted Domain 切换到 Trusted Domain 时系统崩溃。

#### 如何发现
- 调用 SBI 扩展函数返回错误码 `SBI_ERR_DENIED`
- 查看 OpenSBI 日志显示 `domain X not accessible`
- GDB 显示陷入 `sbi_trap_handler` 的异常处理

#### 根本原因

OpenSBI Domain 机制要求严格的权限检查：

```c
// OpenSBI 源码中 domain 切换的条件
if (current_domain != target_domain && 
    !sbi_domain_is_assigned_hart(target_domain, current_hart)) {
    return SBI_ERR_DENIED;
}
```

**Domain 切换只能在 assigned hart 上进行**。

#### 解决方案

1. **使用正确的 HART 进行 Domain 操作**：确保 CPU7 只运行 Trusted Domain 代码

2. **通过共享内存进行跨域通信**，而非直接切换 Domain：
   ```c
   // Trusted/Untrusted 共享内存区域
   #define SHM_BASE 0xBFF00000
   
   struct msg_queue {
       volatile uint32_t head;
       volatile uint32_t tail;
       uint8_t data[SHM_SIZE];
   };
   ```

3. **实现 SBI 调用代理**：Untrusted Domain 通过 SBI 调用请求 Trusted Domain 执行操作

### 难题五：任务调度与时钟中断竞争

#### 问题描述
OS（Untrusted Domain）使用 RTC 中断（IRQ 11）进行调度，但 Trusted Domain 的 FreeRTOS 使用 CLINT 定时器。**两者可能产生竞争条件**。

#### 如何发现
- 任务调度延迟不稳定
- 时钟中断丢失（观察 `scause` 寄存器）
- 两个 domain 同时运行时某一方卡住

#### 根本原因

```c
// os/src/trap.c
case 5: // Supervisor timer interrupt (CLINT)
    set_next_trigger();
    schedule();
```

**RTC 中断（外部中断）和 CLINT 定时器中断（本地中断）是独立的**，但 OS 只处理了 CLINT 中断。

#### 解决方案

1. **统一使用 CLINT 进行调度**（修改 RTC 驱动，改用 `mtimecmp`）

2. **正确处理 RTC 中断**：
   ```c
   // 在 trap_handler 中添加
   case IRQ_S_EXT: // 外部中断
       uint32_t irq = plic_claim();
       if (irq == RTC_IRQ) {
           rtc_handler();
       }
       plic_complete(irq);
       break;
   ```

### 难题七：物理内存布局硬编码导致扩展困难

#### 问题描述

#### 如何发现
- 调整固件大小时链接错误或运行时崩溃
- 新增固件组件时需要修改 start.s、链接脚本、设备树

```asm
// boot/start.s

slli    a1, a1, 20      // a1 = 0x80000000 (OpenSBI dest)
```

### 神坑九：物理页号(PPN)与虚拟地址(VA)直接强制转换导致内存越界/缺页崩溃

#### 问题描述
从编译日志中可以看到大量严重警告，例如：
```
warning: initialization of 'TrapContext *' from 'u64' makes pointer from integer without a cast (在 src/task.c 中)
```

在进行进程切换（fork 或 exec）加载上下文时，系统突然触发 **Load/Store Page Fault (13或15号异常)**，导致 Kernel Panic，QEMU 卡死或自动重启。

#### 如何发现该问题

**运行期现象**：
- 系统会打印陷入信息 `scause = 0xd` 或 `0xf`，`stval` 指向一个奇怪的地址（比如非常小的值，或者未映射的物理地址）

**调试期现象**：
- 在 GDB 中单步执行 `TrapContext* cx_ptr = np->trap_cx_ppn;` 下方的指针解引用代码时直接跑飞

#### 根本原因

在 RISC-V 的分页机制（如 Sv39）开启后，CPU 执行指令使用的都是**虚拟地址**。而页表中存储的、以及页框分配器返回的是**物理页框号 (PPN)**。PPN 只是物理地址除以 4096 的一个编号。

**直接把一个 `u64` 的 PPN 当作 C 语言指针去访问**，CPU 会把这个 PPN 值当成虚拟地址去查页表，这必然查不到或者查错！

#### 解决方案

封装地址转换宏，把 PPN 转成物理地址（`* 4096`），然后再转成内核可用的虚拟地址（如果内核有高半核映射，需要加上地址偏移）：

```c
// 地址转换宏示例
#define PAGE_SIZE       4096
#define PAGE_SIZE_BITS  12
#define PPN_TO_PA(ppn)  ((ppn) << PAGE_SIZE_BITS)

// 假设内核虚拟地址 = 物理地址 + KERNEL_BASE_OFFSET
#define PA_TO_KVA(pa)   ((pa) + KERNEL_BASE_OFFSET)
#define PPN_TO_KVA(ppn) (PA_TO_KVA(PPN_TO_PA(ppn)))

// 使用示例
TrapContext* cx_ptr = (TrapContext*)PPN_TO_KVA(np->trap_cx_ppn);
```

### 神坑十：修改页表后忘记刷新 TLB (Translation Lookaside Buffer)

#### 问题描述
在 `task.c` 和 `syscall.c` 中实现 `fork`、`exec` 或者 `sbrk` 时，代码逻辑看起来完美无缺，页表也确实写入了物理内存，但是用户程序跑起来却报 **Instruction Page Fault**（执行未映射内存）或者读到旧的脏数据。

更诡异的是，这个问题可能是**随机发生**的，有时好有时坏。

#### 如何发现该问题

- 在 GDB 中，当你怀疑缺页时，手动查看内存 `x /x 虚拟地址`，发现 GDB 能读到数据（因为 GDB 绕过了部分 CPU 缓存），但 CPU 实际执行却报缺页
- 查看 `satp` 寄存器值确实是对的，但在页表生效的边界处发生了崩溃

#### 根本原因

RISC-V 硬件为了加速地址转换，会将 **虚拟地址->物理地址** 的映射缓存在 TLB 中。当内核修改了内存中的页表（PTE）时，**硬件 TLB 不会自动更新**。CPU 依然使用内存缓存里的旧映射，导致访问出错。

#### 解决方案

在每次切换进程地址空间（写入 `satp` 寄存器）之后，或者修改了已存在映射的页表项之后，必须插入一条特定的汇编指令：

```c
// RISC-V SFENCE.VMA 指令刷新 TLB
static inline void sfence_vma(void) {
    __asm__ __volatile__("sfence.vma zero, zero" ::: "memory");
}

// 在切换页表后调用
void switch_page_table(PageTable* pt) {
    uint64_t satp_val = (8UL << 60) | (pt->root_ppn);
    __asm__ __volatile__("csrw satp, %0" :: "r"(satp_val));
    sfence_vma();  // 必须刷新 TLB！
}
```

### 神坑十一：VirtIO 磁盘的并发数据读取死锁 (Virtio Block I/O Deadlock)

#### 问题描述
从代码结构看，正在实现 `virtio_disk.c` 和 `bio.c`。当 OS 尝试从磁盘加载 `user_shell` ELF 文件时，整个系统**完全挂起**，也不报异常，也不往下执行。敲键盘没反应，终端输出停滞。

#### 如何发现该问题

- 在 VSCode 中按下 F5 暂停 QEMU（Pause），观察左侧的调用栈（Call Stack）
- 极大概率会发现 PC 指针停留在 `bio.c` 或 `virtio_disk.c` 的一个类似 `while(!b->valid) { sleep(b); }` 的死循环/自旋锁中

#### 根本原因

VirtIO 磁盘的操作是**异步**的。内核向 VirtIO 控制器发送读取命令后，当前进程必须挂起睡眠。当磁盘数据准备好后，它会触发一个**外部中断 (External Interrupt)**。

如果遇到死锁，原因通常有三个：

1. **PLIC**（平台级中断控制器）没有正确配置把 VirtIO 的中断路由给 S 态
2. 内核在等待期间**关闭了 S 态全局中断** (`sstatus.SIE=0`)，导致磁盘中断永远进不来
3. 中断处理函数 `trap.c` 中响应了外部中断，但没有调用 `virtio_disk_intr()` 去唤醒对应的等待进程

#### 解决方案

1. **确认 PLIC 配置正确**：
   ```c
   // 初始化时设置 VirtIO 中断优先级和使能
   plic_set_priority(VIRTIO_IRQ, 1);
   plic_set_threshold(0);  // S-mode 阈值
   plic_enable_interrupt(VIRTIO_IRQ);
   ```

2. **确认 trap.c 的外部中断分支中**，通过 PLIC claim 读取到的 IRQ 号如果对应 VirtIO，必须调用唤醒函数：
   ```c
   // trap.c 中的外部中断处理
   case IRQ_S_EXT:
       int irq = plic_claim();
       if (irq == VIRTIO0_IRQ) {
           virtio_disk_intr();  // 唤醒等待的进程
       }
       plic_complete(irq);
       break;
   ```

3. **检查在调用驱动读写函数前**，是否错误地持有自旋锁同时又关了中断。确保在等待 IO 返回时，让出 CPU 并允许中断发生：
   ```c
   // 错误示例：持有锁 + 关中断会导致死锁
   // acquire(&disk.lock);
   // virtio_disk_rw(buf);
   // while(!buf->done) { sleep(buf); }  // 中断进不来，永远等不到！
   
   // 正确做法：等待时释放锁，允许中断
   virtio_disk_rw(buf);
   while(!buf->done) { 
       sleep(buf, &disk.lock);  // 睡眠时释放锁
   }
   ```

### 神坑十二：上下文切换时陷入死循环或寄存器状态破坏 (User/Kernel Stack 错乱)

#### 问题描述
用户态进程陷入内核（系统调用 `sys_write` 等）后，内核执行完毕调用 `sret` 返回，但系统直接崩溃。或者第二个进程刚切换上去就抛出 **Load Access Fault**。

#### 如何发现该问题

- 此时单步调试 C 代码没用，必须在 **GDB Disassembly (反汇编)** 视图下，紧盯 `entry.S` 中的 `_alltraps` 和 `_trapret` 汇编
- 断点打在 `sret` 指令，执行前检查 `sp`（栈指针）和 `sepc`（返回地址）。如果 `sp` 指向了错误的内存，或者 `sepc` 指向了内核地址，一定会崩

#### 根本原因

用户态和内核态必须使用**隔离的栈**。在陷阱入口处（`_alltraps`），通常需要先把用户栈指针保存到某个地方，然后把内核栈指针恢复出来（通常借助 `csrrw sp, sscratch, sp`）。

如果在内核执行期间发生了嵌套陷阱（内核态下又发生了页错误或时钟中断），`sscratch` 可能被意外覆盖，或者栈平衡被破坏，导致寄存器恢复时读取到全是乱码的内存。

#### 解决方案

1. **严格区分来自 U 态的 Trap 和来自 S 态的 Trap**：
   - `entry.S` 处理 U->S 的陷阱入口
   - `kerneltrap.S` 处理 S->S 的陷阱入口
   - **必须确保它们不要串线！**

2. **在 C 语言的 `trap_handler` 中，第一件事务必检查 `sstatus.SPP` 位**：
   ```c
   void trap_handler() {
       uint64_t sstatus;
       __asm__ __volatile__("csrr %0, sstatus" : "=r"(sstatus));
       int from_user = (sstatus & SSTATUS_SPP) == 0;
       
       if (from_user) {
           // 来自用户态的处理
       } else {
           // 来自内核态的处理（可能是内核 bug 导致的异常）
           panic("Kernel trap!");
       }
       // ...
   }
   ```

3. **确保 `sscratch` 寄存器的正确使用**：
   ```asm
   # entry.S: U-mode -> S-mode 入口
   _alltraps:
       # 交换 sp 和 sscratch，此时 sp = 内核栈，sscratch = 用户栈
       csrrw sp, sscratch, sp
       # 保存所有寄存器到内核栈...
       
   _trapret:
       # 恢复所有寄存器...
       # 交换回来，此时 sp = 用户栈，sscratch = 内核栈
       csrrw sp, sscratch, sp
       sret
   ```

### 神坑十三：设备树 boot-hart 配置缺失导致 Domain 初始化失败

#### 问题描述
OpenSBI 启动日志显示 `Domain0: boot hart id not found`，Trusted Domain 或 Untrusted Domain 无法正常初始化，系统只能启动一个 Domain 或完全挂起。

#### 如何发现该问题

- OpenSBI 启动时打印错误信息：`Domain0: boot hart id not found`
- 只有部分 HART 能正常启动，其他 HART 进入错误状态
- 查看设备树节点时发现缺少关键属性

#### 根本原因

设备树中 `opensbi-domains` 节点下的 domain 配置缺少 `boot-hart` 字段，OpenSBI 无法确定哪个 HART 是启动核：

```dts
// 错误的配置
tdomain: trusted-domain {
    compatible = "opensbi,domain,instance";
    possible-harts = <&cpu7>;
    // 缺少 boot-hart 字段！
    next-addr = <0x0 0xBF800000>;
    next-mode = <0x1>;
};
```

OpenSBI 的 Domain 机制要求每个 domain 必须明确指定 boot hart，用于初始化该 domain 的上下文。

#### 解决方案

**修正设备树配置，添加 boot-hart 字段**：

```dts
// 正确的配置
tdomain: trusted-domain {
    compatible = "opensbi,domain,instance";
    possible-harts = <&cpu7>;
    boot-hart = <&cpu7>;          // 必须指定 boot hart
    next-addr = <0x0 0xBF800000>; // Trusted Domain 入口
    next-mode = <0x1>;            // S-Mode
    regions = <&tmem0 0x7>, <&tmem1 0x7>, ...;  // 内存区域权限
};

udomain: untrusted-domain {
    compatible = "opensbi,domain,instance";
    possible-harts = <&cpu0 &cpu1 &cpu2 &cpu3 &cpu4 &cpu5 &cpu6>;
    boot-hart = <&cpu0>;          // 指定 CPU0 为 boot hart
    next-addr = <0x0 0x80200000>;
    next-mode = <0x1>;
    regions = <&tmem0 0x0>, ...;  // 排除 trusted 区域
};
```

### 神坑十四：FreeRTOS 与 OpenSBI 中断处理冲突

#### 问题描述
Trusted Domain 的 FreeRTOS 任务不切换，或切换到错误任务后崩溃。时钟中断似乎被忽略，任务调度完全失效。

#### 如何发现该问题

- FreeRTOS 启动后能运行第一个任务，但无法切换到其他任务
- 使用 GDB 查看 `mtimecmp` 寄存器，发现定时器中断从未触发
- 对比 OpenSBI 启动前后 CLINT 寄存器状态，发现配置被修改

#### 根本原因

1. **中断向量表地址冲突**：FreeRTOS 使用自己的中断向量表，但 OpenSBI 已经设置了 `stvec`
2. **CLINT 定时器配置被覆盖**：OpenSBI 初始化时配置了 CLINT，但 FreeRTOS 启动时又重新初始化
3. **任务栈大小不足**：默认 `configMINIMAL_STACK_SIZE` 太小，导致上下文切换时栈溢出
4. **特权模式不匹配**：FreeRTOS RISC-V 端口默认运行在 M-Mode，但 Trusted Domain 运行在 S-Mode

#### 解决方案

1. **调整 FreeRTOS 配置**（`FreeRTOSConfig.h`）：
   ```c
   #define configUSE_PREEMPTION                    1
   #define configCPU_CLOCK_HZ                      10000000
   #define configTICK_RATE_HZ                      1000
   #define configMINIMAL_STACK_SIZE                1024  // 增大栈大小
   ```

2. **修改端口代码适应 S-Mode**（`portASM.S`）：
   ```asm
   // 原始代码（M-Mode）
   csrr t0, mstatus
   // 修改为 S-Mode
   csrr t0, sstatus
   
   // 原始代码
   csrw mtvec, t0
   // 修改为
   csrw stvec, t0
   ```

3. **使用 OpenSBI 提供的定时器中断代理**：
   ```c
   // 在 FreeRTOS 中通过 SBI 调用设置定时器
   sbi_set_timer(next_trigger_time);
   ```

### 神坑十五：compile_commands.json 合并失败导致 VSCode 代码索引失效

#### 问题描述
VSCode 中 clangd 无法正确索引代码，代码跳转和补全功能失效，开发效率严重下降。项目包含多个子模块（os、opensbi-1.2、trusted_domain、qemu），每个都有自己的编译数据库。

#### 如何发现该问题

- VSCode 中无法跳转到函数定义
- 头文件包含路径错误，显示红色波浪线
- 查看 Output 面板中的 clangd 日志，显示找不到 compile_commands.json
- 某些子项目的 compile_commands.json 根本没有生成

#### 根本原因

1. **bear 工具生成范围有限**：默认只捕获顶层构建命令，子项目构建过程未被记录
2. **路径格式不统一**：不同子项目生成的 json 文件中路径格式不一致（绝对路径 vs 相对路径）
3. **合并脚本错误**：最初的 `build-with-bear.sh` 没有正确处理缺失的 json 文件
4. **qemu 编译特殊**：qemu 使用 meson 构建系统，bear 无法直接捕获

#### 解决方案

**改进 build-with-bear.sh 脚本**：

```bash
#!/bin/bash
SHELL_FOLDER=$(cd "$(dirname "$0")";pwd)

# 先执行完整构建并捕获命令
bear --output ${SHELL_FOLDER}/compile_commands.json -- bash ${SHELL_FOLDER}/build.sh

# 子项目列表
SUB_DIRS=("os" "opensbi-1.2" "trusted_domain")

for dir in "${SUB_DIRS[@]}"; do
    if [ -f "${SHELL_FOLDER}/${dir}/compile_commands.json" ]; then
        echo "Merging ${dir}/compile_commands.json..."
        jq -s 'add' "${SHELL_FOLDER}/compile_commands.json" \
            "${SHELL_FOLDER}/${dir}/compile_commands.json" > temp.json
        mv temp.json "${SHELL_FOLDER}/compile_commands.json"
    else
        echo "Warning: $dir/compile_commands.json not found, skipping..."
    fi
done

# 特殊处理 qemu（手动生成或从 build 目录复制）
if [ -f "${SHELL_FOLDER}/qemu-8.0.2/build/compile_commands.json" ]; then
    echo "Merging qemu compile_commands.json..."
    jq -s 'add' "${SHELL_FOLDER}/compile_commands.json" \
        "${SHELL_FOLDER}/qemu-8.0.2/build/compile_commands.json" > temp.json
    mv temp.json "${SHELL_FOLDER}/compile_commands.json"
fi

rm -f temp.json
echo "compile_commands.json merged successfully!"
```

### 神坑十六：VSCode 调试配置反复失效

#### 问题描述
VSCode 无法启动调试，或调试时无法正确附加到 QEMU。修改代码后调试配置经常失效，需要手动调整 tasks.json 和 launch.json。

#### 如何发现该问题

- 按 F5 启动调试时，提示 "无法连接到 GDB"
- QEMU 已经启动但 VSCode 无法附加
- 修改 OS 代码后，调试时运行的还是旧版本
- 调试 QEMU 本身时代码跳转不正确

#### 根本原因

1. **构建任务顺序错误**：task.json 中的构建命令没有正确编译 OS 模块
2. **工作目录不一致**：调试器和被调试程序的工作目录不匹配
3. **QEMU 启动参数错误**：`-s -S` 参数位置或使用方式不正确
4. **多项目调试配置冲突**：同时调试 OS 和 QEMU 时需要不同的配置

#### 解决方案

**迭代演进的 tasks.json 配置**：

```json
// 阶段一：简单的 build 命令（初始）
{
    "label": "build",
    "type": "shell",
    "command": "${workspaceFolder}/build.sh"
}

// 阶段二：分离编译步骤（发现未编译 OS）
{
    "label": "build timeros",
    "type": "shell",
    "command": "cd ${workspaceFolder}/os && make clean && make && cd ${workspaceFolder} && ${workspaceFolder}/build.sh"
}

// 阶段三：支持调试版本的最终配置（修复）
{
    "label": "build timeros",
    "type": "shell",
    "command": "${workspaceFolder}/build.sh && cd ${workspaceFolder}/os && make && cd ${workspaceFolder}
}
```

**launch.json 关键配置**：

```json
{
    "name": "Debug OS",
    "type": "gdb",
    "request": "attach",
    "target": "localhost:1234",
    "remote": true,
    "cwd": "${workspaceFolder}/os",
    "valuesFormatting": "parseText"
}
```

## 🔧 总结建议

**当前迫切需要做的是先清理编译警告**。不要忽视那些隐式声明和类型强转的警告，在 64 位寻址的裸机开发中，**整数和指针的隐式转换是引发 "薛定谔的 Bug" 的罪魁祸首！**

建议启用更严格的编译选项：
```makefile
CFLAGS += -Wall -Wextra -Werror=int-to-pointer-cast -Werror=pointer-to-int-cast
```
