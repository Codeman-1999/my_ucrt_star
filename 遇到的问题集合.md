# my_ucrt_star 项目开发难题分析

## 📋 项目架构概览

```
启动流程: MROM → LowLevelBoot → OpenSBI → [Trusted Domain (CPU7, FreeRTOS) + Untrusted Domain (CPU0-6, OS)]
                    ↓
            设备树配置多域隔离
```

## 🔴 难题一：多核启动时序与同步问题

### 问题描述
OpenSBI 初始化 8 个 HART，但不同域的启动地址不同（Trusted: 0xBF800000, Untrusted: 0x80200000）。**非启动核（non-boot hart）可能在固件还没准备好时就尝试跳转**。

### 如何发现
- 串口输出只有 CPU0 或 CPU7 的打印，其他核无输出
- QEMU 挂起，需要 GDB 连接到不同 HART 查看 PC 寄存器
- OpenSBI 日志显示 `hart X: domain startup failed`

### 根本原因

```asm
; boot/start.s 中的忙等待代码
csrr    a0, mhartid
li      t0, 0x0     
beq     a0, t0, _no_wait
loop    0x1000    ; 非0核固定延时，可能不够
```

非 boot hart 的延时固定，如果 OpenSBI 初始化慢于延时，会导致跳转时目标代码未就绪。

### 解决方案

1. **使用 OpenSBI 提供的 HART 间同步机制**（IPI 或共享内存标志位）

2. **在目标代码入口处设置同步屏障**：
   ```c
   // 每个域的入口代码
   static volatile int ready_flag = 0;
   
   void domain_entry() {
       if (hartid == boot_hart) {
           // 初始化代码...
           __sync_synchronize();
           ready_flag = 1;
       } else {
           while (!ready_flag) { wfi(); }
       }
   }
   ```

## 🔴 难题二：Trust Domain 内存隔离失效

### 问题描述
Trusted Domain (0xBF800000-0xBFFFFFFF) 和 OS 内存（0x80000000-0xC0000000）有重叠区域。**OS 可能意外访问或覆盖 Trusted Domain 内存**。

### 如何发现
- Trusted Domain 随机崩溃或数据损坏
- QEMU 内存检查器显示非预期的写操作
- OpenSBI 触发 PMP（Physical Memory Protection）异常

### 根本原因

查看设备树配置：

```dts
// quard_star_sbi.dts
udomain: untrusted-domain {
    regions = <&tmem0 0x0>, <&tmem1 0x0>, ... <&allmem 0x7>;
    // allmem 覆盖整个 0x0-0xFFFFFFFFFFFFFFFF，权限 0x7(RWX)
};
```

`allmem` 的权限设置允许 Untrusted Domain 访问所有内存，**隔离策略失效**。

### 解决方案

1. **修正设备树中的内存区域权限**：
   ```dts
   udomain: untrusted-domain {
       regions = <&tmem0 0x0>, <&tmem1 0x0>, ...; // 排除 trusted 区域
       // 不要包含 allmem，或将其权限设为 0
   };
   ```

2. **启用 PMP 检查**，在 OpenSBI 配置中确保 `PMP_GRANULARITY` 设置正确

## 🔴 难题三：PLIC 中断路由冲突

### 问题描述
UART2（Trusted Domain 专用串口）的中断可能被错误地路由到 Untrusted Domain，或反之。

### 如何发现
- Trusted Domain 的串口无中断响应（查询方式可以工作）
- PLIC 寄存器读取显示中断被错误核心认领
- 使用 QEMU `-d int` 查看中断分发日志

### 根本原因

设备树中所有 UART 都连接到同一个 PLIC，但 Trusted/Untrusted Domain 的中断上下文是独立的：

```dts
plic@c000000 {
    interrupts-extended = <&cpu0_intc 11>, <&cpu0_intc 9>, ...>; // 所有 CPU
};
```

### 解决方案

1. **为 Trusted Domain 配置独立的中断控制器**（如 APLIC/IMSIC，需要 QEMU 支持）

2. **在软件层面过滤中断**：在 trap handler 中检查当前 domain，忽略不属于本 domain 的中断

3. **修改 QEMU 的 PLIC 实现**，支持 domain-based 中断屏蔽（需要修改 `hw/intc/sifive_plic.c`）

## 🔴 难题四：OpenSBI Domain 切换失败

### 问题描述
调用 `sbi_domain_reset()` 或尝试从 Untrusted Domain 切换到 Trusted Domain 时系统崩溃。

### 如何发现
- 调用 SBI 扩展函数返回错误码 `SBI_ERR_DENIED`
- 查看 OpenSBI 日志显示 `domain X not accessible`
- GDB 显示陷入 `sbi_trap_handler` 的异常处理

### 根本原因

OpenSBI Domain 机制要求严格的权限检查：

```c
// OpenSBI 源码中 domain 切换的条件
if (current_domain != target_domain && 
    !sbi_domain_is_assigned_hart(target_domain, current_hart)) {
    return SBI_ERR_DENIED;
}
```

**Domain 切换只能在 assigned hart 上进行**。

### 解决方案

1. **使用正确的 HART 进行 Domain 操作**：确保 CPU7 只运行 Trusted Domain 代码

2. **通过共享内存进行跨域通信**，而非直接切换 Domain：
   ```c
   // Trusted/Untrusted 共享内存区域
   #define SHM_BASE 0xBFF00000
   
   struct msg_queue {
       volatile uint32_t head;
       volatile uint32_t tail;
       uint8_t data[SHM_SIZE];
   };
   ```

3. **实现 SBI 调用代理**：Untrusted Domain 通过 SBI 调用请求 Trusted Domain 执行操作

## 🔴 难题五：任务调度与时钟中断竞争

### 问题描述
OS（Untrusted Domain）使用 RTC 中断（IRQ 11）进行调度，但 Trusted Domain 的 FreeRTOS 使用 CLINT 定时器。**两者可能产生竞争条件**。

### 如何发现
- 任务调度延迟不稳定
- 时钟中断丢失（观察 `scause` 寄存器）
- 两个 domain 同时运行时某一方卡住

### 根本原因

```c
// os/src/trap.c
case 5: // Supervisor timer interrupt (CLINT)
    set_next_trigger();
    schedule();
```

**RTC 中断（外部中断）和 CLINT 定时器中断（本地中断）是独立的**，但 OS 只处理了 CLINT 中断。

### 解决方案

1. **统一使用 CLINT 进行调度**（修改 RTC 驱动，改用 `mtimecmp`）

2. **正确处理 RTC 中断**：
   ```c
   // 在 trap_handler 中添加
   case IRQ_S_EXT: // 外部中断
       uint32_t irq = plic_claim();
       if (irq == RTC_IRQ) {
           rtc_handler();
       }
       plic_complete(irq);
       break;
   ```

## 🔴 难题六：QEMU 自定义机器类型兼容性问题

### 问题描述
升级 QEMU 版本或切换主机环境后，`quard-star` 机器类型无法识别。

### 如何发现
- 运行 `./run.sh` 报错 `Unknown machine type: quard-star`
- 编译 QEMU 时 `quard_star.c` 编译失败（API 变更）

### 根本原因

自定义 QEMU 机器的代码位于 `qemu-8.0.2/hw/riscv/quard_star.c`，**绑定特定 QEMU 版本**。

### 解决方案

1. **将 QEMU 修改做成 patch 文件**：
   ```bash
   diff -u qemu-8.0.2-original/hw/riscv/quard_star.c qemu-8.0.2/hw/riscv/quard_star.c > quard-star.patch
   ```

2. **使用 QEMU 的插件机制**（QOM）避免修改源码

3. **升级到新版 QEMU 时同步更新 machine 代码**：
   - 检查 `include/hw/boards.h` 的 MachineClass API 变更
   - 更新 `hw/riscv/meson.build` 中的编译规则

## 🔴 难题七：物理内存布局硬编码导致扩展困难

### 问题描述
`boot/start.s` 中内存地址是硬编码的，修改固件位置时需要同步修改多处代码。

### 如何发现
- 调整固件大小时链接错误或运行时崩溃
- 新增固件组件时需要修改 start.s、链接脚本、设备树

### 根本原因

```asm
// boot/start.s
li      a0, 0x202
slli    a0, a0, 20      // a0 = 0x20200000 (OpenSBI source)
li      a1, 0x800
slli    a1, a1, 20      // a1 = 0x80000000 (OpenSBI dest)
```

**地址分散在汇编代码、链接脚本、设备树、QEMU 代码中**。

### 解决方案

1. **创建统一的内存布局头文件**（`include/memory_layout.h`）：
   ```c
   #define FLASH_BASE      0x20000000
   #define OPENSBI_FLASH   0x20200000
   #define OPENSBI_DRAM    0x80000000
   #define TRUSTED_FLASH   0x20400000
   #define TRUSTED_DRAM    0xBF800000
   ```

2. **使用生成脚本**在编译时自动同步所有配置文件

3. **在设备树中定义所有固件位置**，通过 `chosen` 节点传递：
   ```dts
   chosen {
       opensbi-fw = <0x0 0x80200000>;
       trusted-fw = <0x0 0xBF800000>;
   };
   ```

## 🔴 难题八：SBI 调用版本兼容性问题

### 问题描述
Trusted Domain 的 FreeRTOS 调用 SBI 函数时返回 `SBI_ERR_NOT_SUPPORTED`。

### 如何发现
- `sbi_console_putchar()` 无输出
- 返回值检查显示错误码 -2 (`NOT_SUPPORTED`)
- OpenSBI 版本和 SBI 规范版本不匹配

### 根本原因

```c
// trusted_domain/riscv/sbi.c
struct sbiret sbi_ecall(int ext, int fid, uint64_t arg0, ...) {
    // 使用错误的调用约定
}
```

OpenSBI 1.2 实现了 SBI 规范 1.0，但调用方式可能有差异。

### 解决方案

1. **使用 OpenSBI 提供的 `libsbi.a` 进行链接**，而非自己实现 SBI 调用

2. **核对 SBI 规范版本**：
   ```c
   // 在初始化时检查 SBI 版本
   struct sbiret ret = sbi_get_spec_version();
   if (ret.error) {
       // 处理错误
   }
   ```

3. **使用内联汇编确保调用约定正确**：
   ```c
   register long a7 __asm__("a7") = ext;
   register long a6 __asm__("a6") = fid;
   // ...
   __asm__ __volatile__("ecall"
                        : "=r"(a0), "=r"(a1)
                        : "r"(a7), "r"(a6), ...);
   ```

## ⚠️ 神坑九：物理页号(PPN)与虚拟地址(VA)直接强制转换导致内存越界/缺页崩溃

### 问题描述
从编译日志中可以看到大量严重警告，例如：
```
warning: initialization of 'TrapContext *' from 'u64' makes pointer from integer without a cast (在 src/task.c 中)
```

在进行进程切换（fork 或 exec）加载上下文时，系统突然触发 **Load/Store Page Fault (13或15号异常)**，导致 Kernel Panic，QEMU 卡死或自动重启。

### 如何发现该问题

**运行期现象**：
- 系统会打印陷入信息 `scause = 0xd` 或 `0xf`，`stval` 指向一个奇怪的地址（比如非常小的值，或者未映射的物理地址）

**调试期现象**：
- 在 GDB 中单步执行 `TrapContext* cx_ptr = np->trap_cx_ppn;` 下方的指针解引用代码时直接跑飞

### 根本原因

在 RISC-V 的分页机制（如 Sv39）开启后，CPU 执行指令使用的都是**虚拟地址**。而页表中存储的、以及页框分配器返回的是**物理页框号 (PPN)**。PPN 只是物理地址除以 4096 的一个编号。

**直接把一个 `u64` 的 PPN 当作 C 语言指针去访问**，CPU 会把这个 PPN 值当成虚拟地址去查页表，这必然查不到或者查错！

### 解决方案

封装地址转换宏，把 PPN 转成物理地址（`* 4096`），然后再转成内核可用的虚拟地址（如果内核有高半核映射，需要加上地址偏移）：

```c
// 地址转换宏示例
#define PAGE_SIZE       4096
#define PAGE_SIZE_BITS  12
#define PPN_TO_PA(ppn)  ((ppn) << PAGE_SIZE_BITS)

// 假设内核虚拟地址 = 物理地址 + KERNEL_BASE_OFFSET
#define PA_TO_KVA(pa)   ((pa) + KERNEL_BASE_OFFSET)
#define PPN_TO_KVA(ppn) (PA_TO_KVA(PPN_TO_PA(ppn)))

// 使用示例
TrapContext* cx_ptr = (TrapContext*)PPN_TO_KVA(np->trap_cx_ppn);
```

## ⚠️ 神坑十：修改页表后忘记刷新 TLB (Translation Lookaside Buffer)

### 问题描述
在 `task.c` 和 `syscall.c` 中实现 `fork`、`exec` 或者 `sbrk` 时，代码逻辑看起来完美无缺，页表也确实写入了物理内存，但是用户程序跑起来却报 **Instruction Page Fault**（执行未映射内存）或者读到旧的脏数据。

更诡异的是，这个问题可能是**随机发生**的，有时好有时坏。

### 如何发现该问题

- 在 GDB 中，当你怀疑缺页时，手动查看内存 `x /x 虚拟地址`，发现 GDB 能读到数据（因为 GDB 绕过了部分 CPU 缓存），但 CPU 实际执行却报缺页
- 查看 `satp` 寄存器值确实是对的，但在页表生效的边界处发生了崩溃

### 根本原因

RISC-V 硬件为了加速地址转换，会将 **虚拟地址->物理地址** 的映射缓存在 TLB 中。当内核修改了内存中的页表（PTE）时，**硬件 TLB 不会自动更新**。CPU 依然使用内存缓存里的旧映射，导致访问出错。

### 解决方案

在每次切换进程地址空间（写入 `satp` 寄存器）之后，或者修改了已存在映射的页表项之后，必须插入一条特定的汇编指令：

```c
// RISC-V SFENCE.VMA 指令刷新 TLB
static inline void sfence_vma(void) {
    __asm__ __volatile__("sfence.vma zero, zero" ::: "memory");
}

// 在切换页表后调用
void switch_page_table(PageTable* pt) {
    uint64_t satp_val = (8UL << 60) | (pt->root_ppn);
    __asm__ __volatile__("csrw satp, %0" :: "r"(satp_val));
    sfence_vma();  // 必须刷新 TLB！
}
```

## ⚠️ 神坑十一：VirtIO 磁盘的并发数据读取死锁 (Virtio Block I/O Deadlock)

### 问题描述
从代码结构看，正在实现 `virtio_disk.c` 和 `bio.c`。当 OS 尝试从磁盘加载 `user_shell` ELF 文件时，整个系统**完全挂起**，也不报异常，也不往下执行。敲键盘没反应，终端输出停滞。

### 如何发现该问题

- 在 VSCode 中按下 F5 暂停 QEMU（Pause），观察左侧的调用栈（Call Stack）
- 极大概率会发现 PC 指针停留在 `bio.c` 或 `virtio_disk.c` 的一个类似 `while(!b->valid) { sleep(b); }` 的死循环/自旋锁中

### 根本原因

VirtIO 磁盘的操作是**异步**的。内核向 VirtIO 控制器发送读取命令后，当前进程必须挂起睡眠。当磁盘数据准备好后，它会触发一个**外部中断 (External Interrupt)**。

如果遇到死锁，原因通常有三个：

1. **PLIC**（平台级中断控制器）没有正确配置把 VirtIO 的中断路由给 S 态
2. 内核在等待期间**关闭了 S 态全局中断** (`sstatus.SIE=0`)，导致磁盘中断永远进不来
3. 中断处理函数 `trap.c` 中响应了外部中断，但没有调用 `virtio_disk_intr()` 去唤醒对应的等待进程

### 解决方案

1. **确认 PLIC 配置正确**：
   ```c
   // 初始化时设置 VirtIO 中断优先级和使能
   plic_set_priority(VIRTIO_IRQ, 1);
   plic_set_threshold(0);  // S-mode 阈值
   plic_enable_interrupt(VIRTIO_IRQ);
   ```

2. **确认 trap.c 的外部中断分支中**，通过 PLIC claim 读取到的 IRQ 号如果对应 VirtIO，必须调用唤醒函数：
   ```c
   // trap.c 中的外部中断处理
   case IRQ_S_EXT:
       int irq = plic_claim();
       if (irq == VIRTIO0_IRQ) {
           virtio_disk_intr();  // 唤醒等待的进程
       }
       plic_complete(irq);
       break;
   ```

3. **检查在调用驱动读写函数前**，是否错误地持有自旋锁同时又关了中断。确保在等待 IO 返回时，让出 CPU 并允许中断发生：
   ```c
   // 错误示例：持有锁 + 关中断会导致死锁
   // acquire(&disk.lock);
   // virtio_disk_rw(buf);
   // while(!buf->done) { sleep(buf); }  // 中断进不来，永远等不到！
   
   // 正确做法：等待时释放锁，允许中断
   virtio_disk_rw(buf);
   while(!buf->done) { 
       sleep(buf, &disk.lock);  // 睡眠时释放锁
   }
   ```

## ⚠️ 神坑十二：上下文切换时陷入死循环或寄存器状态破坏 (User/Kernel Stack 错乱)

### 问题描述
用户态进程陷入内核（系统调用 `sys_write` 等）后，内核执行完毕调用 `sret` 返回，但系统直接崩溃。或者第二个进程刚切换上去就抛出 **Load Access Fault**。

### 如何发现该问题

- 此时单步调试 C 代码没用，必须在 **GDB Disassembly (反汇编)** 视图下，紧盯 `entry.S` 中的 `_alltraps` 和 `_trapret` 汇编
- 断点打在 `sret` 指令，执行前检查 `sp`（栈指针）和 `sepc`（返回地址）。如果 `sp` 指向了错误的内存，或者 `sepc` 指向了内核地址，一定会崩

### 根本原因

用户态和内核态必须使用**隔离的栈**。在陷阱入口处（`_alltraps`），通常需要先把用户栈指针保存到某个地方，然后把内核栈指针恢复出来（通常借助 `csrrw sp, sscratch, sp`）。

如果在内核执行期间发生了嵌套陷阱（内核态下又发生了页错误或时钟中断），`sscratch` 可能被意外覆盖，或者栈平衡被破坏，导致寄存器恢复时读取到全是乱码的内存。

### 解决方案

1. **严格区分来自 U 态的 Trap 和来自 S 态的 Trap**：
   - `entry.S` 处理 U->S 的陷阱入口
   - `kerneltrap.S` 处理 S->S 的陷阱入口
   - **必须确保它们不要串线！**

2. **在 C 语言的 `trap_handler` 中，第一件事务必检查 `sstatus.SPP` 位**：
   ```c
   void trap_handler() {
       uint64_t sstatus;
       __asm__ __volatile__("csrr %0, sstatus" : "=r"(sstatus));
       int from_user = (sstatus & SSTATUS_SPP) == 0;
       
       if (from_user) {
           // 来自用户态的处理
       } else {
           // 来自内核态的处理（可能是内核 bug 导致的异常）
           panic("Kernel trap!");
       }
       // ...
   }
   ```

3. **确保 `sscratch` 寄存器的正确使用**：
   ```asm
   # entry.S: U-mode -> S-mode 入口
   _alltraps:
       # 交换 sp 和 sscratch，此时 sp = 内核栈，sscratch = 用户栈
       csrrw sp, sscratch, sp
       # 保存所有寄存器到内核栈...
       
   _trapret:
       # 恢复所有寄存器...
       # 交换回来，此时 sp = 用户栈，sscratch = 内核栈
       csrrw sp, sscratch, sp
       sret
   ```

---

## 🔧 总结建议

在 VSCode 中，绝大多数这类神坑都可以通过 **GDB 查看汇编指令** 和 **查看 CSR 寄存器状态** 发现。

**当前迫切需要做的是先清理编译警告**。不要忽视那些隐式声明和类型强转的警告，在 64 位寻址的裸机开发中，**整数和指针的隐式转换是引发 "薛定谔的 Bug" 的罪魁祸首！**

建议启用更严格的编译选项：
```makefile
CFLAGS += -Wall -Wextra -Werror=int-to-pointer-cast -Werror=pointer-to-int-cast
```
